最稳的一句话结论

Safest one-line judgment

这份源码更像一个已经成熟上线的核心运行时，外面包着一大圈受 gate、 preview、enterprise 和 build-variant 控制的功能层。

数字口径

Count Hygiene

这些数字适合放在首页或摘要里，因为它们的统计方式比较稳定，不像“500+ runtime gates”那样高度依赖口径。

89 唯一的 all-caps feature('FLAG') 符号

适合表达“变体压力很大”。 Stable headline metric for variant pressure.

941 feature('...') 调用点

更接近大家口中“这个仓库到处都是 gate”时真正想表达的量级。 Closer to the intuitive "heavily gated" reading.

86 命令面 / command surfaces

比单纯数 slash commands 更稳。 Broader and more reliable than counting slash commands only.

42 工具目录 / tool directories

能反映广度，但不等于“47+ agent tools”这种海报口径。 Useful breadth signal, not the same as "agent tool" counts.

证据最强的功能面

Strongly Supported Feature Families

这部分是“海报里写出来基本站得住”的内容。即使它们有些仍然受 gate 控制，也确实不是几条字符串或半成品 stub。

需要保留意见的说法

Claims That Need Caveats

这部分不是说“错”，而是说如果想写成严谨的技术结论，就不能像海报那样一笔带过。

表述过强 / Over-read

Daemon

源码里确实有 assistant daemon mode、daemon worker、cron/SDK daemon 相关内容， 但它并不足以证明一个通用、成熟、对外公开的后台常驻服务模式。 The direction is real; the poster wording is too confident.

只能部分验证 / Partially inspectable

Self-Hosted Runner

cli.tsx 里有真实入口，但这份 snapshot 缺少导入的完整实现目录。 The entry path is real, but the full subsystem is not inspectable here.

统计口径问题 / Metric caveat

500+ Runtime Gates

这类数字只有在先定义“runtime gate”算什么之后才有意义。 Without a counting rule, this is not a stable technical metric.

不是源码结论 / Not a code claim

Leak / Discovery Story

泄漏来源、谁先发现，属于 provenance 问题，不是源码结构本身能证明的结论。 Keep this in attribution notes, not in the feature verdicts.
