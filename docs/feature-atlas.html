Claude Code Source Capability Map

What this snapshot clearly implements, what it gates, and what remains partial

This page is a compact reading of the committed src/ tree only. It groups visible capability surfaces by evidence strength and avoids making release-status, provenance, or marketing claims that the code itself cannot prove.

  • Runtime spine
    Persistence, model loop, tools, remote execution, and UI are deeply implemented.
  • Specialized surfaces
    Many notable families are real, but gated, scoped, or variant-driven.
  • Evidence boundary
    Missing imported subsystems and non-code questions stay outside the verdict.

What this snapshot clearly shows

The tree contains a product-scale runtime with session persistence, tool orchestration, remote execution, extension plumbing, and a large amount of variant-driven capability surface.

What still needs restraint

Some entry paths are real while their imported implementations are missing from this snapshot. Build flags, platform constraints, and entitlements also mean “implemented” does not automatically mean “universally on”.

Method

The page is intentionally conservative. It only says what the committed tree can directly support and keeps adjacent speculation out of the capability inventory.

Scope

only the committed bundle

Everything here is derived from the checked-in snapshot under src/, plus repo-local counts used elsewhere in this bundle.

Evidence Rule

implemented, gated, partial

Implemented means executable code or wired UI/service paths. Gated means the surface exists but is controlled by flags, variants, entitlements, or platform scope. Partial means the entrypoint exists but the full subsystem is not fully inspectable here.
